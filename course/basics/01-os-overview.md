# 操作系统全景

这门课的核心活动是对照 Linux 内核源码理解操作系统原理，然后用 Zig 从零构建操作系统组件。我们从最基本的问题开始。

打开终端，运行 `ps aux | head`：

```
$ ps aux | head
USER       PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
root         1  0.0  0.1 169436 13092 ?        Ss   Mar01   0:12 /sbin/init
root         2  0.0  0.0      0     0 ?        S    Mar01   0:00 [kthreadd]
root         3  0.0  0.0      0     0 ?        I<   Mar01   0:00 [rcu_gp]
user      1234  0.2  1.5 345678 12345 pts/0    Ss   10:00   0:03 /bin/bash
user      2345  1.5  3.2 567890 23456 pts/0    S+   10:05   0:10 vim main.c
user      3456  0.0  0.5 234567  5678 pts/1    S+   10:10   0:01 python3 server.py
...
```

几百个进程同时运行，共用一颗 CPU（或几颗）、一块物理内存、一块磁盘。每个进程都以为自己独占了整台机器，互不干扰。谁在管理这件事？

答案是**操作系统**。操作系统是硬件和应用程序之间的中间层，负责管理硬件资源并为程序提供统一的抽象接口。操作系统管理的对象是硬件，所以我们需要先理解它管理的是什么：处理器怎么组织、内存有几层、设备怎么和 CPU 通信，这些构成了**计算机体系结构**的核心。操作系统本身也是一个程序，它的代码怎么组织？是把所有功能放在一个大程序里，还是拆成独立模块通过消息通信？这个选择定义了**内核结构**。内核结构不是随意选的，它遵循一组反复出现的**设计原则**，比如机制与策略分离、最小权限。这些原则也不是凭空发明的，而是从几十年的实践中提炼而来。回顾**操作系统历史**，每一代系统都在解决上一代的瓶颈，解决方案逐渐沉淀为今天的设计原则。

## 操作系统

操作系统(Operating System, OS)是管理计算机硬件资源、并为应用程序提供服务接口的系统软件。

这个定义包含两层职责。第一层是**资源管理**：CPU 时间怎么分配给各个进程，物理内存怎么划分，磁盘空间怎么组织。开篇的 `ps aux` 显示了几百个进程同时运行，但 CPU 核心可能只有 4 个。操作系统的调度器决定每个时刻哪个进程占用哪个核心，每隔几毫秒就做一次切换，速度快到用户感觉所有程序在同时运行。

第二层是**抽象**：应用程序不需要知道硬盘是 SSD 还是机械盘，不需要知道网卡的具体型号。操作系统把硬件差异藏在统一的接口背后。对程序来说，写文件就是调用 `write()`，不管底层是什么设备。读网络数据也是 `read()`，不管底层是以太网还是 WiFi。"一切皆文件"是 UNIX 系统（Linux 的前身）最核心的抽象思想：文件、设备、管道、网络连接，统统通过文件描述符访问，使用相同的 `read`/`write`/`close` 接口。

操作系统有多无处不在？用 `strace` 追踪一个最简单的命令 `ls`：

```
$ strace ls 2>&1 | head -20
execve("/usr/bin/ls", ["ls"], 0x7ffd... /* 63 vars */) = 0
brk(NULL)                               = 0x563f...
openat(AT_FDCWD, "/etc/ld.so.cache", O_RDONLY|O_CLOEXEC) = 3
mmap(NULL, 92599, PROT_READ, MAP_PRIVATE, 3, 0) = 0x7f2a...
close(3)                                = 0
openat(AT_FDCWD, "/lib/x86_64-linux-gnu/libc.so.6", O_RDONLY|O_CLOEXEC) = 3
read(3, "\177ELF...", 832)              = 832
mmap(NULL, 2037344, PROT_READ, MAP_PRIVATE|MAP_DENYWRITE, 3, 0) = 0x7f2a...
close(3)                                = 0
...
write(1, "file1.txt\nfile2.txt\n", 20)  = 20
close(1)                                = 0
exit_group(0)                           = ?
```

`strace` 显示了程序和操作系统之间的每一次交互。`ls` 做的事情看起来很简单：列出目录内容并打印到终端。但 `ls` 没有能力直接读磁盘，也没有能力直接往屏幕上写字符。每一步都要请求操作系统代为完成：`openat` 打开文件，`read` 读取数据，`write` 输出到终端，`mmap` 映射共享库到内存。即使是最简单的命令，背后也有几十次与操作系统的交互。

我们来写一个最小的 C 程序，让它通过操作系统在终端打印一行文字：

```c
// hello.c
#include <unistd.h>

int main(void) {
    write(1, "hello\n", 6);
    return 0;
}
```

```
$ gcc hello.c -o hello && ./hello
hello
```

`write(1, "hello\n", 6)` 告诉操作系统："往文件描述符 1（标准输出）写入 6 个字节"。程序自己不知道标准输出连接的是终端模拟器、串口还是网络套接字。操作系统负责把这 6 个字节送到正确的地方。这就是抽象的力量：程序只需要知道"写到 fd 1"，其余的细节由操作系统处理。

:::thinking 如果没有操作系统会怎样？
假设没有操作系统，每个程序都要自己管理硬件。一个文本编辑器需要自己写磁盘驱动来保存文件，自己写显卡驱动来显示文字，自己写键盘驱动来接收输入。

更严重的问题是多个程序的共存。如果两个程序同时运行，它们怎么分配 CPU 时间？怎么避免一个程序覆盖另一个程序的内存？怎么防止一个程序直接读取另一个程序的数据？

没有操作系统，每个程序员都必须重新实现这些基础功能，而且程序之间没有任何隔离保护。一个程序的 bug 可以摧毁整个系统中所有正在运行的程序。操作系统的存在让程序员可以专注于业务逻辑，把硬件管理和资源隔离交给一个经过充分测试的公共层。
:::

## 计算机体系结构

计算机体系结构(computer architecture)是计算机系统的硬件组织方式，包括处理器、存储器和 I/O 设备的结构及其交互方式。

操作系统的很多设计决策直接来源于硬件的物理特性。理解体系结构中影响操作系统设计的部分，是后续所有章节的基础。

**处理器组织**。最简单的计算机只有一个处理器核心，所有程序轮流使用这一个核心。现代计算机通常有多个核心，它们的组织方式直接影响操作系统的调度策略。用 `lscpu` 可以查看当前机器的处理器拓扑：

```
$ lscpu
Architecture:           x86_64
CPU(s):                 8
Thread(s) per core:     2
Core(s) per socket:     4
Socket(s):              1
NUMA node(s):           1
L1d cache:              32K
L1i cache:              32K
L2 cache:               256K
L3 cache:               8192K
NUMA node0 CPU(s):      0-7
```

这台机器有 1 个物理插槽(socket)，4 个物理核心(core)，每个核心通过超线程(Hyper-Threading)提供 2 个逻辑线程，总共 8 个逻辑 CPU。操作系统的调度器把进程分配到这 8 个逻辑 CPU 上运行。

多核处理器有两种主要的组织方式：

**对称多处理(Symmetric Multiprocessing, SMP)** 是所有核心共享同一块物理内存，每个核心访问任意内存地址的延迟相同。大多数消费级多核 CPU 都是 SMP 架构。操作系统的调度器可以把进程调度到任意核心上运行，不需要关心内存位置。

**非一致性内存访问(Non-Uniform Memory Access, NUMA)** 是每个处理器（或处理器组）有自己的本地内存。访问本地内存很快，访问其他处理器的远端内存则要慢得多。服务器和高性能计算系统通常采用 NUMA 架构。操作系统的调度器需要尽量把进程调度到它的数据所在的 NUMA 节点上，减少跨节点内存访问。

**存储层次**。CPU 执行指令的速度远快于内存提供数据的速度。如果 CPU 每次都要等内存返回数据，大量时间会浪费在等待上。解决方案是在 CPU 和主存之间插入多级容量递增、速度递减的缓存(cache)：

| 层级 | 典型容量 | 典型延迟 | 说明 |
|------|---------|---------|------|
| 寄存器 | 几十到几百字节 | < 1 ns | CPU 内部，速度最快 |
| L1 缓存 | 32-64 KB（每核心） | ~1 ns | 分为数据缓存(L1d)和指令缓存(L1i) |
| L2 缓存 | 256 KB - 1 MB（每核心） | ~3-4 ns | 每个核心私有 |
| L3 缓存 | 8-32 MB（共享） | ~10-12 ns | 同一插槽内所有核心共享 |
| 主存(RAM) | 8-512 GB | ~50-100 ns | 所有核心共享，掉电丢失 |
| SSD | 256 GB - 数 TB | ~10-100 μs | 持久存储，比 RAM 慢约 1000 倍 |
| HDD | 1-20 TB | ~3-10 ms | 机械磁盘，比 SSD 慢约 100 倍 |

从寄存器到 HDD，延迟跨越了 7 个数量级。这个巨大的差距决定了操作系统的很多设计。页缓存(page cache)把频繁访问的磁盘数据缓存在内存中，避免每次都去读慢速磁盘。虚拟内存(virtual memory)在物理内存不足时把不常用的页面换出到磁盘，为当前活跃的程序腾出空间。这些机制的本质都是利用存储层次的速度差异，把热数据尽量放在离 CPU 近的位置。

:::expand 为什么需要缓存层次
缓存层次之所以有效，是因为程序的内存访问具有局部性(locality)。局部性分为两种：

**时间局部性(temporal locality)**：刚访问过的数据很可能在短时间内再次被访问。循环体中的变量就是典型例子：每次迭代都读写同一个计数器。

**空间局部性(spatial locality)**：访问某个地址后，附近的地址很可能很快被访问。遍历数组时，访问了 `a[0]` 之后大概率接着访问 `a[1]`、`a[2]`。

CPU 缓存利用这两种局部性工作。当 CPU 访问一个地址时，硬件不只加载这一个字节，而是把包含它的整个缓存行(cache line，通常 64 字节)一起加载到缓存中。这样，如果程序接下来访问相邻的地址（空间局部性），数据已经在缓存里了。如果程序很快再次访问同一个地址（时间局部性），数据也在缓存里。

局部性不是运气，而是程序结构的必然结果：代码是顺序执行的（指令具有空间局部性），循环会反复执行同一段代码（指令具有时间局部性），数据结构通常在内存中连续存放（数据具有空间局部性）。
:::

**总线与 DMA**。CPU、内存和 I/O 设备之间通过总线(bus)连接。现代系统的总线已经不再是单一的共享通道，而是由多种互联结构组成（如 PCIe 连接显卡和 NVMe SSD，内存控制器直接集成在 CPU 内部），但概念上它们的职责相同：在组件之间搬运数据。

如果每次数据传输都要 CPU 亲自参与（逐字节从设备复制到内存），CPU 就会被 I/O 操作占满，无暇执行计算任务。DMA(Direct Memory Access，直接内存访问)解决了这个问题：CPU 只需要告诉 DMA 控制器"从设备 X 搬 N 字节到内存地址 Y"，DMA 控制器就会自主完成数据传输。传输完成后，DMA 控制器通过中断通知 CPU。这样 CPU 在数据传输期间可以继续执行其他任务。

## 内核结构

内核(kernel)是操作系统中运行在最高权限级别的核心程序，直接管理硬件资源并为上层软件提供服务。

操作系统的功能很多：进程管理、内存管理、文件系统、设备驱动、网络协议栈。这些功能怎么组织成一个可运行的程序？不同的组织方式产生了不同的内核结构。

**宏内核(monolithic kernel)** 把所有操作系统功能编译进同一个大程序中，运行在同一个地址空间里。进程管理、内存管理、文件系统、设备驱动、网络协议栈全部在内核态执行，函数之间通过普通的函数调用通信。Linux 就是宏内核。

宏内核的优点是性能：所有组件在同一个地址空间中，互相调用不需要跨越地址空间的开销。文件系统要读磁盘，直接调用磁盘驱动的函数即可。缺点是隔离性差：任何一个内核组件的 bug 都可能破坏整个内核的内存。一个有问题的设备驱动可以导致整个系统崩溃。

**微内核(microkernel)** 只在内核态保留最核心的功能：进程调度、进程间通信(IPC)和基本的内存管理。文件系统、设备驱动、网络协议栈等其余功能都作为独立的用户态进程运行。这些进程之间通过消息传递(message passing)通信。L4 和 QNX 是微内核的代表。

微内核的优点是隔离性：设备驱动运行在用户态，一个驱动崩溃不会影响内核和其他驱动，系统可以自动重启故障组件。缺点是性能：原来宏内核里一次函数调用能完成的事，在微内核中变成了用户态进程之间的消息传递。

用一个具体场景说明这个开销差异。假设用户程序要读一个文件。在宏内核(Linux)中，`read()` 系统调用进入内核后，内核代码直接调用 VFS 层的函数，VFS 调用具体文件系统的函数，文件系统调用块设备驱动的函数。整个过程是一连串的函数调用，没有地址空间切换。在微内核中，用户程序发消息给文件系统进程，文件系统进程再发消息给磁盘驱动进程，磁盘驱动进程操作硬件后把数据通过消息返回给文件系统进程，文件系统进程再把数据返回给用户程序。每条消息都要经过内核中转（用户态→内核态→用户态），一次文件读取涉及至少 4 次地址空间切换，累积的 IPC 开销可能很显著。

**混合内核(hybrid kernel)** 试图兼顾两者。macOS 的 XNU 内核基于微内核 Mach，但把大量功能（包括 BSD 子系统、文件系统、网络栈）放回了内核态以减少 IPC 开销。Windows NT 内核也是类似的设计：架构上是微内核风格（分层、模块化），但大部分服务运行在内核态。

:::expand 外核与可加载内核模块
**外核(exokernel)** 是一种学术研究中的极端设计。它的思想是：内核不提供任何抽象，只负责安全地复用硬件资源。每个应用程序通过库操作系统(library OS)自己实现文件系统、网络栈等功能。外核的优势是应用程序可以根据自己的需求优化资源使用策略，但缺点是每个应用程序的开发复杂度大幅增加。MIT 的 Aegis/ExOS 是外核的代表性研究项目，它没有进入工业界使用。

Linux 虽然是宏内核，但支持**可加载内核模块(Loadable Kernel Module, LKM)**。LKM 让内核可以在运行时动态加载和卸载功能模块（主要是设备驱动），不需要重新编译整个内核。用 `lsmod` 可以查看当前已加载的模块：

```
$ lsmod | head
Module                  Size  Used by
snd_hda_codec_hdmi     73728  1
snd_hda_intel          53248  0
iwlwifi               421888  1 iwlmvm
ext4                  802816  1
```

LKM 不改变 Linux 的宏内核本质：加载后的模块运行在内核态、与内核共享地址空间，拥有完全的内核权限。LKM 的好处是灵活性（不需要为每个硬件组合编译不同的内核），不是隔离性。
:::

:::thinking Linux 选了宏内核，是不是就不稳定？
宏内核的理论缺陷是明确的：任何内核组件的 bug 都可能破坏整个系统。但 Linux 在实践中非常稳定，原因是多方面的。

首先，Linux 有庞大的开发者社区和严格的代码审查流程。每个进入主线的补丁都经过 maintainer 审核。其次，Linux 提供了多种内核态调试和防护机制：KASAN（内核地址消毒器）检测内存越界，lockdep 检测死锁和锁序违反，UBSAN 检测未定义行为。这些工具在开发和测试阶段捕获了大量潜在问题。

最后，实践表明内核稳定性问题的主要来源是设备驱动，而不是核心子系统。驱动代码量占内核源码的大部分，质量参差不齐。Linux 通过驱动模型（`struct device`、`struct driver`）和统一的框架约束驱动的行为，降低了驱动 bug 影响内核核心的概率。

宏内核与微内核的争论曾经非常激烈（1992 年 Linus Torvalds 和 Andrew Tanenbaum 的公开辩论是其中最著名的一次）。几十年过去，工业界的事实是：主流通用操作系统（Linux、Windows、macOS）都选择了宏内核或混合内核，微内核主要用在对可靠性有极端要求的嵌入式场景（QNX 用于汽车和航空系统）。性能开销在通用场景下是决定性因素。
:::

## 设计原则

操作系统的设计原则是指导内核架构和接口设计的一组核心思想。

这些原则反复出现在内核的各个子系统中。掌握它们，能帮助理解具体机制背后的设计动机。

**机制与策略分离**。机制回答"怎么做"：提供一种能力或操作方式。策略回答"做什么"：决定何时、对谁使用这种能力。把两者分开，意味着改变策略不需要修改机制的实现。

Linux 内核的进程调度是一个典型例子。机制是上下文切换(context switch)：保存当前进程的寄存器状态，恢复下一个进程的寄存器状态，切换页表。无论选哪个进程来运行，这个切换过程都是一样的。策略是调度算法：从所有就绪进程中选择下一个运行的进程。Linux 通过 `sched_class` 结构体定义了调度策略的接口：

```c
// kernel/sched/sched.h (simplified)
struct sched_class {
    void (*enqueue_task)(struct rq *rq, struct task_struct *p, int flags);
    void (*dequeue_task)(struct rq *rq, struct task_struct *p, int flags);
    struct task_struct *(*pick_next_task)(struct rq *rq);
    ...
};
```

不同的调度策略实现这同一组接口：CFS（完全公平调度器）适合普通进程，RT（实时调度器）适合延迟敏感的任务，Deadline 调度器适合有截止时间要求的任务。增加一种新的调度策略，只需要实现 `sched_class` 接口，不需要修改上下文切换的代码。

**最小权限(principle of least privilege)**。每个组件只应拥有完成其任务所必需的最小权限。权限越大，出错时的破坏范围越大。

这条原则在硬件层面的体现是 CPU 的特权级(privilege level)。x86-64 处理器定义了 4 个特权级（Ring 0 到 Ring 3），Linux 使用其中两个：内核运行在 Ring 0（最高权限，可以执行所有指令、访问所有硬件），用户程序运行在 Ring 3（最低权限，不能直接访问硬件、不能执行特权指令）。用户程序如果试图执行特权指令（比如直接读写 I/O 端口），CPU 会触发异常，内核接管处理。这条硬件机制保证了用户程序不可能绕过操作系统直接操作硬件。下一课会详细讨论这个机制。

即使在内核内部，Linux 也通过 Capabilities 机制细化了权限控制。传统 Unix 的权限模型只有两级：普通用户和 root。root 用户拥有所有权限，这违反了最小权限原则。一个 Web 服务器需要绑定 80 端口（小于 1024 的端口只有 root 能绑定），在传统模型下只能以 root 身份运行。但 root 还意味着可以读写任意文件、杀死任意进程、加载内核模块。一旦这个 Web 服务器被攻破，攻击者就获得了整个系统的控制权。

Linux Capabilities 把 root 的权限拆分成几十个独立的能力(capability)。常见的 Capabilities 包括：

| Capability | 允许的操作 |
|-----------|-----------|
| `CAP_NET_BIND_SERVICE` | 绑定小于 1024 的端口 |
| `CAP_SYS_PTRACE` | 跟踪任意进程（调试器需要） |
| `CAP_KILL` | 向任意进程发送信号 |
| `CAP_NET_RAW` | 使用原始套接字（ping 需要） |
| `CAP_DAC_OVERRIDE` | 忽略文件权限检查 |

回到 Web 服务器的例子：只需要授予 `CAP_NET_BIND_SERVICE` 这一个能力，服务器就能以普通用户身份绑定 80 端口，同时不具备其他 root 权限。即使被攻破，攻击者能做的事也非常有限。

## 操作系统历史

操作系统的历史可以看作一条不断解决性能瓶颈的链条。每一代系统都在回答上一代遗留的问题。理解这条链条，就能理解今天操作系统为什么长成这个样子。

**问题：CPU 大部分时间在空转。** 1950 年代的计算机没有操作系统。一台计算机一次运行一个程序，程序员把打孔卡片交给操作员，操作员把卡片装进读卡器，启动程序，等程序运行完再取出结果，然后装入下一个程序。在装卸卡片、检查结果的过程中，价格昂贵的 CPU 完全闲置。一台 IBM 704 每小时的运行成本超过 200 美元（1950 年代的美元），而人工装卸卡片的间隔可能长达数分钟。

**解决方案：批处理系统(batch processing)。** 把多个作业(job)排成队列，存储在磁带上。监控程序驻留在内存中，一个作业结束后自动从磁带加载下一个，不需要操作员干预。CPU 不再等待人工操作。GM-NAA I/O（1956 年）是最早的批处理系统之一，运行在 IBM 704 上。

但新的问题随之出现。

**问题：单个作业独占 CPU，I/O 等待时浪费计算资源。** 批处理解决了作业之间的间隔问题，但在一个作业内部，CPU 仍然会空闲。一个作业在等待磁带读写或打印机输出时，CPU 只能空转。I/O 设备的速度比 CPU 慢几个数量级，一个典型的科学计算作业可能有 80% 的时间在等 I/O。

**解决方案：多道程序(multiprogramming)。** 内存中同时加载多个作业。当正在运行的作业等待 I/O 时，操作系统切换到另一个就绪的作业继续执行。CPU 在多个作业之间来回切换，利用率大幅提升。IBM OS/360（1964 年）是第一个广泛使用的多道程序操作系统。

多道程序引入了两个新的技术需求。第一是 CPU 调度：内存中有多个作业就绪，选哪个运行？第二是内存保护：多个作业共享物理内存，必须防止一个作业读写另一个作业的数据。这两个需求催生了操作系统中最核心的两个子系统：调度器和内存管理。

**问题：程序员无法与运行中的程序交互。** 批处理系统中，程序员提交作业后只能等结果。如果程序有 bug，要等几个小时才能看到错误输出，修改后重新提交再等几个小时。一天可能只能跑两三次调试，开发效率极低。

**解决方案：分时系统(time-sharing)。** 多个用户通过终端同时连接到一台计算机。操作系统给每个用户分配一小段 CPU 时间（时间片，通常几十毫秒），在用户之间快速轮转，让每个用户都感觉自己独占整台机器。MIT 的 CTSS(Compatible Time-Sharing System, 1961)是第一个演示分时概念的系统。

分时系统要求更精细的调度算法（时间片多长？哪个用户优先？）和更完善的内存保护（用户数量远多于批处理的作业数量，内存隔离更复杂），还要求快速的上下文切换，因为每隔几十毫秒就要换一个用户。

**问题：Multics 太复杂了。** CTSS 的成功激发了更大的野心。MIT、贝尔实验室和通用电气在 1964 年联合启动了 Multics(Multiplexed Information and Computing Service)项目，目标是构建一个功能完备的分时操作系统，支持上百个用户同时在线。Multics 在安全性和功能性上做了很多创新：分段内存(segmented memory)管理、硬件环保护(ring protection，8 个特权级)、层级文件系统(hierarchical file system，今天所有操作系统的文件系统都源于这个设计)、动态链接(dynamic linking)。但项目规模远超预期，开发进度严重滞后，代码复杂度失控。贝尔实验室在 1969 年退出了这个项目。

**解决方案：UNIX。** 退出 Multics 后，贝尔实验室的 Ken Thompson 和 Dennis Ritchie 决定保留 Multics 的好想法，但用截然不同的方式实现。UNIX（1969 年首个版本运行在 PDP-7 上）的设计哲学是简洁：每个程序做一件事并做好，程序之间通过管道(pipe)组合。层级文件系统保留了 Multics 的设计，但环保护简化为两级（用户态和内核态）。Dennis Ritchie 随后设计了 C 语言并用它重写了 UNIX，这让 UNIX 成为第一个主要由高级语言编写的操作系统，可以相对容易地移植到不同硬件上。

UNIX 的核心概念（进程、fork/exec、文件描述符、管道、层级文件系统）至今仍是操作系统教学和实践的基础。本课程讨论的几乎所有机制都起源于 UNIX。

**问题：UNIX 分裂。** AT&T（贝尔实验室的母公司）将 UNIX 源码授权给大学和企业后，各自发展出不同的分支：BSD(Berkeley Software Distribution)加入了 TCP/IP 网络和虚拟内存，System V(AT&T)走另一条路线，商业公司推出了 SunOS、HP-UX、AIX、IRIX 等各自的变体。这些变体 API 不完全兼容，同一个程序要在不同 UNIX 上运行需要修改和重新编译。POSIX 标准试图统一接口，但各厂商实现程度不一。更大的障碍是商业授权：商业 UNIX 的费用让个人用户和小型组织无法负担。

**解决方案：Linux。** 1991 年，芬兰赫尔辛基大学学生 Linus Torvalds 在学习 Andrew Tanenbaum 的教学操作系统 MINIX（微内核设计）后，从零开始编写了一个宏内核。Linux 从第一天起就采用 GPL 许可证发布全部源码，任何人都可以自由使用、修改和分发。开源模式吸引了全球开发者参与，弥补了单人项目在资源上的不足。Linux 迅速成长为功能最完善的开源操作系统内核：服务器市场超过 90% 的份额，全球前 500 名超级计算机全部运行 Linux，Android 手机的底层也是 Linux 内核。本课程对照的正是 Linux 内核源码。

## 小结

| 概念 | 说明 |
|------|------|
| 操作系统(OS) | 管理硬件资源、为应用程序提供抽象接口的系统软件 |
| 资源管理与抽象 | OS 的两个核心职责：分配硬件资源，屏蔽硬件差异 |
| SMP / NUMA | 两种多处理器组织方式，SMP 内存延迟一致，NUMA 有本地/远端之分 |
| 存储层次 | 寄存器→缓存→RAM→SSD→HDD，延迟跨越 7 个数量级 |
| 局部性(locality) | 程序访问内存的时间和空间规律，缓存有效的根本原因 |
| DMA | 设备直接搬运数据到内存，不需要 CPU 逐字节参与 |
| 宏内核 / 微内核 / 混合内核 | 内核代码组织方式：全部在内核态 / 核心在内核态其余在用户态 / 折中 |
| 机制与策略分离 | 机制回答"怎么做"，策略回答"做什么"，两者独立变化 |
| 最小权限 | 每个组件只拥有完成任务所需的最小权限 |
| Ring 0 / Ring 3 | CPU 特权级，内核运行在 Ring 0，用户程序运行在 Ring 3 |

操作系统做两件事：管理硬件资源和提供抽象接口。管理是为了让多个程序安全地共享硬件，抽象是为了让程序员不必关心硬件细节。从批处理到分时到 UNIX 到 Linux，操作系统的每一次演进都在更好地完成这两件事。
